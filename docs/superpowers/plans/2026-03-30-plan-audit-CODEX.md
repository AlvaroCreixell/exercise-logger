Overall, the spec-to-phase mapping is strong and the plans are unusually detailed, but I would not call them A-OK yet. This was a document audit, not a runtime app audit: there is no web/ implementation in the repo yet, so I validated internal consistency, repo assumptions, and claimed verification coverage.

Mechanical / Certain

The plans are not directly executable in this repo’s current Windows/PowerShell environment. They consistently use bash-only constructs such as sleep, background jobs with & and kill %1, rm -rf, mkdir -p, /dev/null, head, and heredoc-based commit messages. Representative examples: phase1-scaffolding.md:95, phase1-scaffolding.md:107, phase2-domain-data.md:81. This pattern repeats through all phase plans.
Phase 4’s weighted-bodyweight handling goes stale. logSet() only promotes effectiveType when creating a new row, but the existing-row update path returns earlier, and editSet() and deleteSet() never recompute it. See phase4-session-management.md:1241, phase4-session-management.md:1286, phase4-session-management.md:1314. That does not satisfy the spec’s session-level rule for weighted bodyweight gym-routine-tracker-design.md:287.
The set logging UI is keyed off effectiveType instead of the prescribed block target, and it explicitly gives cardio distance+duration inputs. See master-plan.md:195, phase6-ui.md:975. The spec says routine inputs are determined by the block prescription, and each set block has exactly one target kind gym-routine-tracker-design.md:197, gym-routine-tracker-design.md:615.
The master plan’s target file structure still includes tailwind.config.ts master-plan.md:365, but Phase 1 explicitly says Tailwind v4 uses no tailwind.config.ts phase1-scaffolding.md:178.
The plan intentionally changes loggedSets.instanceLabel from null to "" at persistence time phase2-domain-data.md:1884, phase4-session-management.md:1187, even though the spec’s data model says string | null gym-routine-tracker-design.md:269.
Phase 1 generates 1x1 placeholder PNGs phase1-scaffolding.md:486 while the manifest declares them as 192x192 and 512x512 phase1-scaffolding.md:540. That makes the manifest metadata mechanically false, so installability verification there is not trustworthy.
Phase 7 claims all 16 acceptance scenarios are covered master-plan.md:273, but its “Scenario 10” test explicitly stops at the data prerequisite and says timer behavior is tested elsewhere phase7-backup-polish.md:2773, phase7-backup-polish.md:2856.
The Phase 7 Playwright smoke test does not match its own stated scope or the shared Playwright setup. It hardcodes http://localhost:5173/... phase7-backup-polish.md:3293 even though Phase 1 config uses preview on 4173 phase1-scaffolding.md:763. It also claims to cover log a set, export, and import round-trip master-plan.md:275, phase7-backup-polish.md:3283, but the provided test finishes without logging a set phase7-backup-polish.md:3364 and only checks that the export button exists phase7-backup-polish.md:3380.
Less Certain / Design / Hardening

The master plan forces strictly linear execution master-plan.md:15, master-plan.md:299, while the spec explicitly frames section 17 as parallel/subagent-friendly once shared contracts are frozen gym-routine-tracker-design.md:952. Sequential execution is safer, but it gives up one of the spec’s intended advantages.
Backup import validation is mostly shape-based and exercise-ID-based phase7-backup-polish.md:757, phase7-backup-polish.md:837. It does not appear to enforce cross-record integrity such as settings.activeRoutineId pointing to an imported routine, sessionExercises.sessionId pointing to an imported session, or loggedSets.sessionExerciseId pointing to an imported session exercise. Tightening that makes imports safer, but also stricter and more expensive to implement.
The ExercisePicker disables exercises already present in the session phase6-ui.md:2085, phase6-ui.md:2167. The spec does not clearly require that restriction for extras, so this is a product choice, not an obvious bug.
Net: the doc set is close, and the overall phase decomposition is good, but it still has a handful of concrete spec drifts and some overstated verification claims. I would fix the “certain” items before treating the plan as implementation-ready.
